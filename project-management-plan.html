Engineering Operations Proposal

Moving from disconnected tooling and low visibility to integrated, GitHub-native project management with clear accountability.

Where We Stand

A unified approach to how we plan, track, and ship work — replacing disconnected tools with a single system built around where the work already happens.

Project management lives in ClickUp, disconnected from GitHub where the work actually happens. There is no real-time visibility into output, and work is reconstructed from memory at the end of each sprint. The result is duplicated effort in tracking and a growing gap between planned work and what's actually happening in the codebase.

Current State

  • ✕ ClickUp disconnected from GitHub
  • ✕ Sprint points don't reflect real effort
  • ✕ Manual sprint setup every 2 weeks
  • ✕ No visibility into daily output
  • ✕ Bugs age without owners
  • ✕ Features "done" but not user-validated
  • ✕ Work reconstructed from memory

Proposed State

  • ✓ GitHub is the single source of truth
  • ✓ Shipped features are the metric
  • ✓ Sprints are cumulative and auto-tracked
  • ✓ Dashboard shows real-time activity
  • ✓ Issues triaged weekly with clear owners
  • ✓ Pairs demo user-facing results
  • ✓ Work is visible as it happens

Key Principles for Improvement

The beliefs behind the changes. If we agree on these, the specific tools and processes are just details.

Transparency by Default

All work is visible in one place. No hidden effort, no invisible contributions. If it happened, it shows up.

Work Where the Code Lives

Project management lives in GitHub. No context switching. No syncing between tools. One source of truth.

Ship to Users, Not to Boards

A feature isn't done when the ticket moves. It's done when a user can see it and it works.

Clean Repos, Fast Teams

Bugs and issues are first-class citizens. A clean codebase is the foundation for shipping fast.

Measure Output, Not Estimates

Sprint points are gone. What you merge, demo, and resolve is what counts. Show your work.

Pairs Over Individuals

Two people own every feature. Knowledge sharing, accountability, and better code are built into the process.

What Actually Changes

A point-by-point summary of every concrete change. This is what happens when we start.

1
Tooling

ClickUp is replaced by GitHub Projects

All sprint planning, feature tracking, and issue management moves into GitHub Projects. ClickUp is no longer used for engineering work. One tool, one source of truth.

2
Tooling

All work is tracked in GitHub

Bugs and improvements are tracked as GitHub Issues in the relevant repo. Features are tracked on a separate sprint board (tool to be determined). PRs link directly to the work they deliver. No separate task lists.

3
Process

Sprint points are removed entirely

What you merge, demo, and resolve is what counts. The dashboard tracks real output automatically from GitHub activity.

4
Process

Every feature is assigned to a pair

At sprint planning, each feature gets two owners. They share responsibility from design through deployment. No handoffs, no single points of failure. Both names show on the board.

5
Process

Issues are triaged before features at every sprint start

Monday sprint planning begins with issue review. Open bugs are assigned owners. Stale issues are closed or re-prioritised. Only after issue triage do we assign feature work.

6
Process

Sprints are cumulative — no reset every 2 weeks

Unfinished work carries over automatically. No manual re-creation of tickets. The backlog is a living list that grows and shrinks based on what gets done.

7
Culture

"Done" means a user can see it, not that a ticket moved

A feature is complete when it's merged, deployed, and demonstrable. The Work Showcase on Thursday is where pairs show what they built from a user's perspective.

8
Culture

Issue identification becomes a valued contribution

Finding and reporting bugs is tracked as output, not busywork. The dashboard shows "Issues Identified" alongside "Issues Solved" and "Features Shipped" as equal metrics.

9
Tracking

A live dashboard replaces status update meetings

Sprint progress, team metrics, and individual output are all auto-generated from GitHub data. No one needs to manually update a board. No one needs to ask "what are you working on?"

10
Tracking

Individual contributions are recognised automatically

The Individual Dashboard highlights features shipped, PRs merged, and issues resolved per person over time. No more invisible effort — everyone's work shows up.

11
Process

Meetings are reduced to four per sprint

Sprint Planning (Monday, Week 1), Progress Check (Monday, Week 2), Work Showcase (Thursday, end of sprint), plus one weekly team meeting per team. Everything else is optional.

12
Culture

Teams own their repos end-to-end

Each team is responsible for the health of their repositories — open issue count, PR review turnaround, and code quality. Repo health is a team metric, not just an individual one.

Proposed Dashboard Design

A single live view for everyone. No digging through ClickUp, no asking "what did you work on?" — auto-generated from GitHub activity. Four views, one source of truth.

Sprint Cycle & Meeting Rhythm

Two-week sprints with a clear rhythm. Minimal meetings, maximum signal.

Sprint Cycle

Week 1 is about starting right. Week 2 is about finishing strong.

Week 1 — Build

  • Monday: Sprint Planning — assign feature pairs, triage issues
  • Team standups: Deep dive into approach and architecture
  • Pairs begin feature implementation
  • Issue fixes worked in parallel with features
  • PRs open early — draft PRs on day 2-3

Week 2 — Ship

  • Monday: Progress Check — surface blockers, adjust scope
  • Features move to review and testing
  • Pairs prepare user-facing demos
  • Issue cleanup — close resolved, re-triage stale
  • Thursday: Work Showcase — demo and celebrate

Meetings

Every meeting has a clear purpose tied to the sprint cycle.

Monday · Week 1

Tech Update: Sprint Planning

Set sprint scope. Assign feature pairs. Review issue backlog. Define what "done" looks like for each feature.

Every 2 weeks
Monday · Week 2

Tech Update: Progress Check

Mid-sprint sync. Are pairs on track? Surface blockers early. Adjust scope if needed. Review open issues.

Every 2 weeks
Thursday · End of Sprint

Work Showcase

Live demo of completed features. Pairs present what they built and how it works from a user perspective. Celebrate shipped work.

Every 2 weeks
Weekly · All Teams

Team Meeting

Every team has a dedicated 1-hour meeting each week. Technical deep-dives, pair coordination, issue triage within their domain.

Weekly · 1 hour per team

Note: Teams may optionally add a short daily standup if they find it useful, but this is not required.
